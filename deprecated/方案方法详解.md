## 1. Getting Started：入门教程

**来源：** `getting-started.ipynb`
**作者：** Bovard（Kaggle 官方）
**定位：** 新手入门、理解游戏机制

### 核心思路

实现了一个最简单的策略——**Nearest Planet Sniper（最近行星狙击手）**：

1. 遍历自己拥有的每颗行星
2. 找到距离最近的**非己方**行星
3. 若当前行星舰船数 ≥ 目标驻军 + 1（最低 20），则发射舰队
4. 发射角度为出发行星到目标行星的 `atan2` 方向

### 关键代码逻辑

```python
ships_needed = max(nearest.ships + 1, 20)
if mine.ships >= ships_needed:
    angle = math.atan2(nearest.y - mine.y, nearest.x - mine.x)
    moves.append([mine.id, angle, ships_needed])
```

### 缺陷分析

| 问题 | 影响 |
|------|------|
| 不计算航行时间 | 目标在舰队航行期间持续生产，到达时舰船可能不足 |
| 所有行星同时发射 | 多颗行星可能瞄准同一目标，浪费舰船 |
| 忽略太阳 | 直穿太阳的舰队被摧毁，完全无效 |
| 无舰队合并/集结 | 后方行星囤积舰船但不支援前线 |
| 忽略彗星 | 对彗星的临时性毫无认知 |

### 适用场景

- 理解游戏 API 和观测/行动格式
- 作为所有其他方案的基线对比
- **不建议直接用于竞赛提交**

## 2. Tactical Heuristic：战术启发式智能体

**来源：** `orbit-wars-2026-tactical-heuristic.ipynb`
**定位：** 高质量启发式算法，单文件提交

### 核心架构

这是一个**全方位的启发式智能体**，实现了完整的游戏物理模拟和策略决策树。代码约 600 行，全部内联在 `submission.py` 中。

### 关键技术

#### 2.1 物理引擎

- **太阳碰撞检测**：圆-线段相交算法，检测舰队路径是否穿越太阳（含 1.5 单位安全余量）
- **太阳绕行**：当直线路径被太阳阻挡时，在太阳两侧尝试 7 种绕行路径点倍率（1.8, 2.3, 3.0, 4.0...），选择最短安全路径
- **舰队速度曲线**：严格匹配游戏引擎的对数速度公式

#### 2.2 移动目标拦截

- **类型分发器**：对静态行星、轨道行星、彗星分别处理位置预测
- **迭代收敛求解**：8 次迭代，收敛容差 0.2（静态）/ 0.4（彗星）
- **彗星安全区**：拒绝距中心 22 单位以内的拦截点（彗星近日点速度过快不值得追逐）

#### 2.3 战斗模拟

- **事件驱动驻军模拟**：按时间排序所有到达的舰队，逐步计算驻军变化
- **到达时所有权判定**（`garrison_at_arrival`）：不仅计算我方到达时的驻军，还考虑在我方之前到达的敌方舰队可能已翻转所有权
- **防御缺口计算**（`defense_shortfall`）：精确计算行星需要多少额外舰船才能在所有攻击波中存活
- **安全可用舰船**（`safe_ships_available`）：计算在考虑所有已到达舰队后行星的最小未来驻军

#### 2.4 舰队追踪

- **几何投影法**：不使用角度比较，而是计算舰队速度向量到行星的投影距离
- **到达时间预测**：通过投影长度/舰队速度计算 ETA

#### 2.5 策略决策

**五阶段回合结构：**

| 阶段 | 名称 | 内容 |
|------|------|------|
| 0 | 解析与分类 | 加载观测，计算游戏状态标志 |
| 1 | 紧急增援 | 计算每颗我方行星的防御缺口，从最近安全行星派遣增援 |
| 2 | 目标优先攻击 | 全局评分所有目标，按评分降序贪心多源分配 |
| 3 | 二次打击 | 使用剩余舰船参与部分覆盖 |
| 4 | 前线集结 | 后方行星向前线转移舰船 |
| 5 | 终局决战 | 最后 ~35 回合全力进攻 |

**目标评分公式（净 ROI）：**
```
score = (production × productive_turns × multipliers) / (needed × (1 + eta × 0.018))
```

**倍率系统：**
- 敌方行星 1.80x（我有所得 + 敌有所失）
- 战略（最高产量）行星 1.25x
- 刚被削弱的敌方行星（检测 `from_planet_id`）追加 0-58% 倍率
- 竞速中立行星（敌人接近中）1.45x
- 前期中立行星 1.50x
- 落后时敌方行星 1.55x
- 4 人局中立行星 1.20x（让敌人互斗）
- 轨道行星 1.05x

**游戏阶段自适应：**
- `is_early (< 70 回合)`：优先抢占中立行星
- `is_late (剩余 < 80)`：减少生产储备，增加攻击性
- `is_very_late (剩余 < 35)`：全押
- `finishing` / `behind` / `trailing` 等战力状态调整
- `multiplayer` 自动检测 2 人 vs 4 人局

#### 2.6 防御系统

- **自适应驻军保留**：`reserve = max(0, defense_shortfall) + production * prod_reserve`
- **后期减少缓冲**：`PROD_RESERVE = 9`（常规）→ `PROD_RESERVE_LATE = 4`（后期）
- **被判定为必失的行星**（`doomed_planets`）：将所有舰船撤离而非死守
- **最小舰队规模**：12 艘（保证足够的航行速度）
- **彗星 ROI 门槛**：剩余生产回合 ≥ 18 才尝试占领

### 优势

- 物理模拟极其精确
- 防御系统成熟
- 多玩家自适应
- 代码高度结构化，模块注释清晰

### 局限

- 启发式参数大量硬编码，场景泛化依赖调参
- 无学习能力
- 舰队路径中与其他行星的意外碰撞未完全考虑

## 3. OW-Proto：评分公式驱动型智能体

**来源：** `orbit-wars-agent-ow-proto-passed-1-000.ipynb`
**作者备注：** 分数峰值 **1080（Top 95）**，稳定 1020-1050（Top 110-130）

### 核心理念

基于一个精心设计的**自定义评分公式**来选择目标，配合协同攻击和防御系统。

### Proto 评分公式

```python
score = (100 - dist) + (15 * t.production) + (10 * enemy_bonus) - (0.7 * total_ships) - (2 * eta)
```

| 项 | 含义 |
|----|------|
| `100 - dist` | 距离越近分越高 |
| `15 × production` | 高产量行星价值高 |
| `10 × enemy_bonus` | 占领敌方行星额外加分（我方获得 + 敌方失去） |
| `-0.7 × total_ships` | 所需舰船越多代价越大 |
| `-2 × eta` | 到达时间越长越不利 |

### 关键技术

#### 3.1 协同攻击系统

当单颗行星无法独立完成占领时：
1. 收集距离目标最近的友方行星（最多 8 颗，`COOP_PLANET_CAP`）
2. 累加舰船直到 ≥ 目标需求
3. 多源同时发射

支持四种协同模式：
- **协同静态无主行星**：直接分配舰船数
- **协同静态有主行星**：对每个攻击源算 ETA，迭代修正产量增长
- **协同移动无主行星**：对每个攻击源解移动拦截角度
- **协同移动有主行星**：最复杂——结合移动轨迹预测 + 产量增长迭代

#### 3.2 防御与增援系统

- **受攻击检测**（`get_planets_under_attack`）：对所有敌方舰队进行 60 回合轨迹投影，检测与友方行星的碰撞
- **移动行星的防御考虑**：为轨道行星计算 60 回合轨迹，用这些位置做碰撞检测
- **增援规划**（`get_reinforcement_plans`）：按攻击波次模拟驻军变化，计算所需增援数量和最晚到达时间
- **舰队轨迹跟踪**（`fleet_trajectories` / `reinforcement_trajectories`）：维护全局状态，每回合更新 ETA

#### 3.3 舰队轨迹管理

```
fleet_trajectories: 记录每支已发射舰队的 (source, target, angle, ships, arrive_tick)
reinforcement_trajectories: 记录每支增援舰队的 (source, target, angle, ships, arrive_tick)
```

- 每回合更新：`arrive_tick -= 1`
- 若舰队不再存在于观测中（已到达/被摧毁），自动清理
- 空闲舰船计算时会扣除已承诺的增援

#### 3.4 移动行星处理

- `fill_moving_planets(obs)`：比较 `planets` 与 `initial_planets` 来识别轨道行星
- `get_planet_trajectories(p, vel)`：生成 60 回合的轨迹预测
- `find_angle_to_moving_planet(p, t, ships, vel)`：找到使舰队在正确时刻抵达移动行星的发射角度
- 移动行星所需的舰船数会额外包含航行期间的产量增长

#### 3.5 行星每回合单舰队限制

每颗行星最多发射 1 支舰队/回合（简单但有效的节制策略）。

### 优势

- 协同攻击机制独特且有效
- 防御系统考虑舰队到达时序
- 对移动和静态行星有分开的处理逻辑
- 评分公式简单直观，易于理解和调参

### 局限

- 增援系统未按行星价值区分优先级
- 增援若不能在敌方到达前赶到则完全放弃——实际可以发送较少舰船（更快）来至少部分救援
- 未考虑与其他行星的意外碰撞
- 评估公式中的权重（100, 15, 10, 0.7, 2）是硬编码的，缺乏理论依据
- 彗星被完全忽略（`if t.id in comet_planet_ids: continue`）

## 4. Reinforcement Learning Tutorial：PPO 强化学习

**来源：** `orbit-wars-reinforcement-learning-tutorial.ipynb`
**定位：** 教学性质的 RL 方案，展示如何用 PPO 训练 Orbit Wars 智能体

### 核心理念

将 Orbit Wars 的每回合决策转化为**监督学习友好**的形式：对每颗拥有的行星，将其决策建模为从候选目标中选择一个（包括"不操作"）。

### 架构设计

#### 4.1 决策建模

- **每个拥有的行星** = 一个独立决策单元
- **动作空间**：从 K 个候选目标中选择 1 个（index 0 = 不操作）
- **舰船数量不通过学习**：使用固定规则 `max(target.ships + 1, 20)`

#### 4.2 候选构建

对每个源行星 `src`：
1. 将敌方/中立/友方行星分别按距离排序
2. 各取 `candidate_count/3` 个（不足则用其他行星填充）
3. 确保 `index 0` 始终为不操作选项

#### 4.3 特征工程（标量特征，非图像）

**self_features（11 维）：**
- 位置 (x, y) 归一化到 [0,1]
- 半径 / 5.0
- 舰船数 / max_ships
- 产量 / max_production
- 是否为轨道行星
- 友方/敌方行星数量比例
- 友方/敌方总舰船比例

**candidate_features（14 维 × K 个候选）：**
- 中立/友方/敌方 one-hot 标志
- 目标位置 + 相对位置
- 距离归一化
- 目标舰船数 / 产量归一化
- 目标是否为轨道行星
- 直线是否穿过太阳
- 源行星的舰船数

**global_features（8 维）：**
- 回合进度比例
- 友方/敌方/中立行星数比例
- 友方/敌方总舰船 + 舰队中的舰船比例

#### 4.4 策略网络

```
self_features ──→ [Linear→ReLU→Linear→ReLU] ──┐
candidate_features(K个) → [Linear→ReLU→Linear→ReLU] ──┤
global_features ──→ [Linear→ReLU→Linear→ReLU] ──┤
                                                  ├──→ 拼接 → target_head → logits
                                                  └──→ 拼接 → value_head → value
```

- 隐藏层大小：128
- 输出：K 个候选的 logits + 1 个状态价值

#### 4.5 训练配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 算法 | PPO | 近端策略优化 |
| 并行环境数 | 2 | 同时运行 2 个游戏环境 |
| Rollout 步数 | 64 | 每次更新收集 64 步 |
| 总更新次数 | 100（演示）/ 2000（完整） | |
| 学习率 | 0.0003 | |
| 裁剪系数 | 0.2 | |
| 折扣因子 | 0.99 | |
| 熵系数 | 0.01 | 鼓励探索 |
| 对手 | 自对弈 | 每 50 次更新同步权重 |

#### 4.6 训练效果

| 训练量 | 对 Sniper 胜率 |
|--------|---------------|
| 0（未训练） | 0% |
| 500 updates（25%） | 30% |
| 1000 updates（50%） | 85% |
| 2000 updates（完整） | 100% |

### 优势

- 纯数据驱动，无需手工设计策略
- 自对弈机制使智能体持续进步
- 特征工程合理
- 完整的训练→评估→演示流程

### 局限

- 动作空间过于简化（舰船数量不学习）
- 候选数量有限（默认 8 个），可能错过远处高价值目标
- 特征不包含舰队轨迹和到达时间等时序信息
- 训练时间较长（2000 次更新需要可观的 GPU 时间）
- 仅支持 2 人局训练

## 5. Robust Agent v4：物理+策略全面升级

**来源：** `orbit-wars-robust-agent.ipynb`
**定位：** 高质量单文件提交，目标 750+ ELO

### 设计哲学

> "物理精确性至关重要" —— 所有计算必须与游戏引擎完全匹配

该方案在战术启发式方案的基础上进行了全面升级，核心改进聚焦于**物理模拟精度**和**策略框架重组**。

### 核心技术

#### 5.1 物理引擎升级

与战术启发式方案相比：
- 迭代次数：4 → 8
- 收敛容差：0.5 → 0.2（静态）/ 0.4（彗星）
- 绕行路径点：3 → 7 种倍率
- 新增**彗星近日点安全检测**：拒绝距中心 22 单位内的拦截点

#### 5.2 策略框架重组

```
阶段 0 → 解析 + 分类
阶段 1 → 紧急防御（保护受威胁行星）
阶段 2 → 目标优先多源攻击（主攻阶段）
阶段 3 → 二次打击（剩余舰船补刀）
阶段 4 → 前线集结（后方 → 前线转移）
阶段 5 → 终局决战（最后 35 回合全力一击）
```

#### 5.3 关键创新

**反击机制（Counter-Attack）：**
- 利用 `from_planet_id` 检测敌方舰队来源
- 刚派出大量舰船的敌方行星更脆弱（检测弱点并优先攻击）

**战略敌人优先（Strategic-Enemy Bonus）：**
- 识别每个敌方玩家的最高产量行星
- 优先摧毁这些战略目标（1.25x 倍率）

**竞速奖励（Race-Condition Bonus）：**
- 当敌方舰队已在飞向某中立行星时，大幅提升其优先级
- 敌人接近度 `enemy_eta / my_eta < 1.35` → 1.45x 倍率

**到达时所有权判定：**
- `garrison_at_arrival(tgt, eta, early_enemy)`：精确模拟敌方舰队先到达导致的行星易主情况

**4 人局自适应：**
- 检测活跃玩家数 ≥ 3 → 进入多人模式
- 集中攻击最弱的敌人（避免两线作战）
- 优先占领中立行星（让其他敌人互斗）

#### 5.4 净 ROI 评分

```
score = (production × productive_turns × multipliers) / (needed × (1 + eta × 0.018))
```

完整的倍率系统（12 种不同倍率条件），涵盖：
- 敌方行星 / 战略价值 / 弱点检测
- 竞速中立 / 阶段偏好 / 战力状态
- 4 人局博弈 / 轨道奖励

### 优势

- 物理模拟是六个方案中最精确的
- 策略框架清晰分层，易于维护和扩展
- 多人博弈有专门的应对策略
- 反击和战略打击使进攻更有针对性

### 局限

- 与战术启发式方案同根，仍有大量硬编码参数
- 协同攻击不如 OW-Proto 灵活
- 无学习能力

## 6. Structured Baseline v11：结构化基线

**来源：** `orbit-wars-structured-baseline.ipynb`
**版本：** v11
**定位：** 最完整的工程化方案，代码量最大（~196KB）

### 设计哲学

> "结构化"体现在：物理→世界模型→策略的清晰三层架构，每个组件可独立测试和替换。

### 核心架构

```
┌──────────────────────────┐
│    🧱 物理引擎 (Physics)   │  ← 速度曲线、碰撞检测、太阳绕行、移动拦截
├──────────────────────────┤
│  🛡️ 世界模型 (World Model) │  ← 舰队投影、战斗模拟、时间线构建
├──────────────────────────┤
│    🤝 策略 (Strategy)      │  ← 目标评分、多源分派、防御、集结
├──────────────────────────┤
│   🛰️ 智能体入口 (Agent)    │  ← 观测解析 + 阶段编排 + 行动输出
└──────────────────────────┘
```

### 关键技术

#### 6.1 物理引擎

与 Robust Agent v4 物理引擎类似，但模块化更好：
- `fleet_speed()` / `segment_hits_sun()` / `safe_route()` 等公用函数
- **边界感知 ETA 模型**：出发点和目标点都在行星表面边界上计算，而非行星中心
- **多次扫描太阳绕行路径点**：找到最早可行的安全拦截窗口

#### 6.2 世界模型

**舰队投影（Fleet Projection）：**
- 不使用角度比较（角度在航行中不变但目标行星位置可变）
- 使用射线-圆命中计时：`t = proj_distance / speed`
- 投影到行星的垂直距离 < 行星半径 + 余量 → 命中

**战斗模拟（Combat Resolution）：**
- 完全匹配游戏引擎的同回合战斗规则
- 按拥有者聚合 → 前两名对消 → 幸存者 vs 驻军

**时间线（Timeline）：**
- 为每个行星构建排序的事件序列
- 防御、占领和撤离决策都查询同一时间线
- 这一设计避免了重复计算并保证一致性

#### 6.3 策略阶段

**Phase 1: 紧急防御**
- 识别脆弱行星（`defense_shortfall > 0`）
- 从 65 单位范围内的安全行星增援
- 仅当行星挽救 ROI > 0 时才尝试救援

**Phase 2: 主攻阶段**
- 目标优先贪心分配（先全局评分，再按优先级分配来源）
- 使用 `committed_to` 字典跟踪对每个目标的承诺舰船数
- 对每个目标，按距离排序来源，依次填满需求

**Phase 3: 二次打击**
- 余舰行星独立评估每个目标
- 找到仍需要舰船的最佳目标 → 发射

**Phase 4: 前线集结**
- 计算每个友方行星到最近敌方行星的距离 → "前线距离"
- 后方行星向前线行星转移 62%（常规）/ 78%（后期）
- 通过中间行星转发（不走直达，经由前线友方行星中转）

**Phase 5: 终局决战**
- 最后 35 回合：所有剩余舰船全押
- 简化的评分：`production × (remaining - turns) - distance × 0.3`

#### 6.4 特殊机制

**集群施压（Swarm Pressure）：**
- 当多波敌军舰队瞄准同一行星时，该行星被视为"被集群压制"
- 集群行星的防御优先级提升

**撞击漏洞利用（Crash Exploits）：**
- 利用行星旋转/彗星移动将舰队"扫入"目标——不用精确拦截
- 这在某些情况下可以减少所需航行时间

**救援与夺回（Rescue and Recapture）：**
- 如果行星即将失守且无法挽救，允许其将所有舰船撤至安全行星
- 后期用集结的兵力夺回失去的行星

### 优势

- 代码量最大、工程化程度最高
- 世界模型/时间线设计减少了重复计算
- 每个模块可独立测试
- 指令集更完整（包含救援/夺回/集群/漏洞利用）

### 局限

- 代码复杂度高，入门门槛最高
- 某些高级机制（如撞击漏洞）的正确性在极端情况下可能需要验证
- 同样无学习能力

### 共同的技术趋势

1. **物理精确性**是高分的关键——所有高分方案都精确模拟了舰队速度曲线、碰撞检测和移动轨迹
2. **到达时所有权判定**是高级方案的分水岭——不仅要算需要多少舰船，还要算舰队到达时"谁拥有"目标
3. **防御需要主动**——等到行星受威胁时通常已经来不及了
4. **舰队追踪/承诺记账**——维护全局状态避免了多源协调中的重复攻击
5. **游戏阶段自适应**——前期抢占、中期扩张、后期决战，策略应随阶段变化
